# ADR-003: Nullable Strategy — Option[T]

- **状态**: 已接受 (v2 — 2026-05-19 更新)
- **日期**: 2026-05-15 (初版), 2026-05-19 (修订)
- **提出者**: 架构决策

## 背景

数据库列和查询参数可能为 NULL。生成代码需要在 MoonBit 类型系统中表示可空性。

## 决策

### 1. 结构体字段使用 `Option[T]`

使用 **`Option[T]`** 作为唯一的可空性表示方式。

```moonbit
// NOT NULL column → 直接类型
pub struct Users {
  id: Int            // id INT NOT NULL
  name: String       // name TEXT NOT NULL
  email: Option[String]  // email TEXT (nullable)
}
```

实现映射函数 `map_internal_type_nullable(ir_type, not_null)`：

```
not_null=true  →  map_internal_type(ir_type)        // 原类型
not_null=false →  Option[map_internal_type(ir_type)] // 包裹 Option
```

### 2. Row getter 使用 nullable 变体

Row 提供成对的类型化 getter：

```
不可空: get_int64(pos)       -> Result[Int64, DBError]
可空:   get_nullable_int64(pos) -> Result[Option[Int64], DBError]
```

Decode 方法根据列的 `not_null` 属性选择对应的 getter：
- `NOT NULL` 列 → `row.get_int64(0)?`
- `NULL` 列 → `row.get_nullable_int64(0)?`

## 理由

- **语言原生**：`Option[T]` 是 MoonBit 内置类型，编译器原生支持模式匹配
- **语义清晰**：`Option::None` 明确表示 NULL 语义，与 `null`/`nil` 区分
- **成对 getter**：编译期保证不可空列不会意外返回 None，调用方无需额外 unwrap
- **与 Future 兼容**：若后续引入自定义 `Nullable[T]`，可通过 Type Mapping Layer 替换

## 后果

- 正向：生成代码直接使用标准库类型，无需额外 runtime 支持
- 正向：调用方可使用 `match`/`if-let` 安全处理 NULL
- 正向：不可空 getter 返回 `Result[T, DBError]`（不含 Option），调用方直接获取值或错误
- 负向：嵌套 `Option[Array[T]]` 时语义略冗余（可用 type alias 缓解）
- 负向：不支持 "不可空但有默认值" 模式（可后续通过 Column Options 扩展）

## 替代方案

- **Nullable[T] 自定义类型**：更语义化但需要额外 runtime 类型定义和转换层
- **默认值填充**：NULL 列自动填充零值，失去类型安全性
- **单一 getter 返回 Option**：不可空列也需要调用方 unwrap，不必要地增加样板代码
