# ADR-010: MoonBit 0.1 Compatibility — Match+For Loop Pattern

- **状态**: 已接受
- **日期**: 2026-05-19
- **提出者**: P1-005 实现

## 背景

MoonBit 0.1 编译器有若干限制，导致最初设计的 codegen 模式无法编译：

1. **`?` 操作符不存在** — MoonBit 0.1 不支持 Result 传播操作符（`expr?`），无法使用 `let val = getter()?` 模式
2. **`{|...|}` lambda 语法无效** — MoonBit 0.1 使用 `fn(params) { body }` 而非 TypeScript 风格的 `{|params| body}`
3. **`return` 在 let-binding 中类型不传播** — `let x = match expr { Ok(v) => v; Err(e) => return Err(e) }` 中，`return` 没有 bottom/never 类型，编译器无法正确推断结果类型

## 决策

### 1. 移除 TryOp，改用嵌套 Match

**之前**: `let id = match row.get_int64(0) { Ok(v) => v; Err(e) => return Err(e) }` → 类型推断失败

**之后**: 嵌套 match 链式传播，每条路径都是 `Ok(val) => Ok({ ... })` 或 `Err(e) => Err(e)`：
```
match row.get_int64(0) {
  Ok(id) => match row.get_string(1) {
    Ok(name) => Ok({ id: id, name: name })
    Err(e) => Err(e)
  }
  Err(e) => Err(e)
}
```

### 2. 移除 Lambda 语法，改用 fn(...) + ForLoop

**之前**: `db.query(sql, params)?.map(|row| T::decode(row)).collect()` — 使用 `?` 和 `{|...|}` lambda

**之后**: `match db.query { Ok(iter) => match iter.collect { Ok(rows) => { let result = []; for row in rows { ... } } } }` 

### 3. 新增 Expr 变体

- `Return(Expr)` — 替代 `TryOp` 作为唯一的控制流转义
- `ForLoop(String, Expr, Array[Expr])` — 显式 for 循环 AST 节点
- `Match(Expr, Array[(String, Expr)])` — 保持，但 arm body 使用显式 pattern 字符串

## 理由

- **编译通过**: 所有 272 测试在 MoonBit 0.1 `--target wasm-gc` 下编译通过
- **无 runtime 依赖**: 不需要 MoonBit 编译器新增特性
- **可读性**: 嵌套 match 是函数式编程的标准模式，语义清晰
- **类型安全**: 每条路径都是显式的 Result 传播，类型完全可推断

## 后果

- 正向：所有 codegen 产出的 MoonBit 代码在 MoonBit 0.1 下合法
- 正向：AST 变体减少（TryOp 移除），语义更清晰
- 负向：:many 查询的函数体比 `?` + lambda 版本更长
- 负向：嵌套 match 层次随字段数线性增长（3 字段 = 3 层 match）
- 负向：`ForLoop` + `Return` 的组合在 for 循环体内部使用 return 退出，是命令式而非函数式风格
