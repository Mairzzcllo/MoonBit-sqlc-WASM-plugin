# ADR-001: AST-based Code Generation Strategy

- **状态**: 已接受
- **日期**: 2026-05-14
- **提出者**: 架构决策

## 背景

sqlc WASM 插件需要生成 MoonBit 类型的数据库访问代码。代码生成方式有两种主要选择：字符串拼接或基于 AST（抽象语法树）生成。

## 决策

使用 **MoonBit AST 定义 + Pretty Printer** 方式生成代码。

- 定义 MoonBit 语言的 AST 节点类型（module、struct、enum、fn、import、type_alias 等）
- Code Generator 模块生成 AST 节点，而非直接拼接字符串
- Pretty Printer 负责将 AST 输出为格式正确的 MoonBit 源代码

## 理由

- **类型安全**: AST 节点由类型系统保证结构正确，编译器可捕获结构错误
- **可测试性**: AST 节点可独立单元测试，无需解析生成代码
- **易维护性**: 新增语法特性只需添加新 AST 节点和 emitter 分支
- **可优化**: 可在 AST 层面做转换/优化（如导入去重、dead code elimination）
- **质量控制**: Pretty Printer 统一处理缩进、换行、排序，避免代码风格不一致

## 后果

- 正向：生成代码格式一致，易于调试和审查
- 正向：AST 可复用于后续的语言特性扩展
- 负向：初期开发成本略高于 string-concat 方式
- 负向：需要维护 AST 定义与 MoonBit 语言演进的同步

## 替代方案

- **String concatenation**: 快速实现，但难以维护、测试、保证格式一致
- **Template-based generation**: 灵活性介于两者之间，但 AST 方案在类型安全和可测试性上更优
