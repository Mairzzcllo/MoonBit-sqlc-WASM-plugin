# ADR-002: Runtime Architecture — Concrete Struct + Closure Pattern

- **状态**: 已接受
- **日期**: 2026-05-15
- **提出者**: 架构决策

## 背景

sqlc 生成的 MoonBit 代码需要调用数据库执行查询。需要一个运行时抽象层（Runtime Abstraction Layer）作为生成代码的编译期依赖。该层不包含真实数据库驱动，仅定义接口。

## 决策

### 1. 使用 Concrete Struct + Closure 模式，而非 Trait

MoonBit 0.1.20260512 不支持：
- 泛型 trait 方法（`query_one[T](self, ...)`）
- 以 trait 作为函数参数类型（trait 对象）

因此，Runtime 采用 concrete struct + closure 字段注入：

```moonbit
pub struct DB {
  exec_fn : (String) -> Int
  execrows_fn : (String) -> Int64
}

pub struct Row {
  get_fn : (Int) -> String
}
```

### 2. MVP Runtime 最小职责

| 组件 | 方法 | 职责 |
|------|------|------|
| `DB` | `exec(sql)` | 执行 SQL，返回受影响行数 (Int) |
| `DB` | `execrows(sql)` | 执行 SQL，返回受影响行数 (Int64) |
| `Row` | `get(index)` | 按索引获取列值 (String) |

MVP 明确排除：
- Transaction（begin/commit/rollback）
- Prepared statement
- Connection lifecycle（open/close/pool）
- Query 结果解码（交由生成代码的 `decode` 约定方法）

### 3. 构造函数约定

MoonBit struct 字段为 file-private，跨包构造需 `pub fn new()`：

```moonbit
pub fn DB::new(exec_fn, execrows_fn) -> DB
pub fn Row::new(get_fn) -> Row
```

### 4. Decoder 采用约定而非 Trait

生成类型实现 `decode(Row) -> Self` 静态方法，不强制实现 `Decoder` trait（减少运行时耦合，允许用户自定义解码）。

## 理由

- **MoonBit 语言限制**：0.1 版本不支持 trait 对象和泛型 trait 方法，concrete struct + closure 是目前唯一可行的抽象模式
- **最小依赖**：生成代码仅依赖 runtime 包，无需额外驱动依赖
- **构造函数防御**：file-private 字段防止用户直接修改内部状态，强制通过构造函数初始化

## 后果

- 正向：运行时极其轻量，无动态分发开销
- 正向：用户可自由选择底层数据库驱动（只需提供符合签名 closures）
- 正向：生成代码与运行时之间严格单向依赖
- 负向：closure 模式比 trait 更缺乏类型语义（无显式接口契约）
- 负向：无法在 DB 上直接定义泛型方法（如 `query_one[T]`）

## 替代方案

- **Trait 方案**：理想情况，但 MoonBit 0.1 不支持必要特性
- **Dynamic dispatch**：使用 `Any` 参数，但 MoonBit 0.1 无 `Any` 类型
- **代码生成包含完整驱动**：破坏 plugin-host separation 原则
