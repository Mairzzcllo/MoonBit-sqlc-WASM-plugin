# ADR-015: emit_interface Limitation — MoonBit Trait Object Gap and MockDB Alternative

- **状态**: 已接受
- **日期**: 2026-06-02
- **提出者**: P1-038 (GAP-6 from Phase D architecture review)
- **阻断因素**: MoonBit 0.1 工具链不支持 trait 对象和泛型 trait 方法

## 背景

### sqlc-gen-go 的 emit_interface 功能

sqlc-gen-go 支持 `emit_interface: true` 配置选项，该选项会生成一个 `Querier` 接口，包含所有查询方法的签名：

```go
// sqlc-gen-go generated code
type Querier interface {
    GetUser(ctx context.Context, id int64) (User, error)
    ListUsers(ctx context.Context) ([]User, error)
    CreateUser(ctx context.Context, arg CreateUserParams) (sql.Result, error)
}
```

用户可以通过以下方式实现基于接口的 mock：

```go
// Go interface-based mocking
type MockQuerier struct {
    GetUserFunc func(ctx context.Context, id int64) (User, error)
}

func (m *MockQuerier) GetUser(ctx context.Context, id int64) (User, error) {
    return m.GetUserFunc(ctx, id)
}
```

### 本项目的需求

GAP-6 在 Phase D 架构差距评审中被识别：插件无法生成可 mock 的查询接口。在 sqlc 生态中，`emit_interface` 是测试基础设施的关键部分，允许用户通过接口 mock 隔离数据库依赖进行单元测试。

## 决策

### emit_interface 目前无法实现

MoonBit 0.1 (2026-05-12) 存在以下限制，阻止生成等效的 trait-based 查询接口：

1. **不支持 trait object** (`dyn Trait`) — 无法将 trait 作为类型使用，因此无法声明接受 `Querier` 参数的函数
2. **不支持泛型 trait 方法** — trait 方法不能有独立于 trait 自身的泛型参数
3. **不支持 trait 继承** — 无法定义 `Querier` 包含多 个方法签名后让 `DB` 实现它

虽然 MoonBit 支持 `trait` 作为编译期接口约束（typeclass），但不能像 Go interface 那样在运行时通过接口类型进行动态分发。

**技术示例 — 以下模式在 MoonBit 0.1 中不可行：**

```moonbit
// 假设想生成的 trait — 在 MoonBit 0.1 中无法使用 trait object
trait Querier {
  query_users_by_id(Int64) -> Result[Option[User], DBError]
  query_users_by_name(String) -> Result[Array[User], DBError]
}

// 无法声明接受 Querier trait object 的函数
fn process_users(db: Querier) -> Result[Array[User], DBError]  // 编译错误！
```

### 替代方案：MockDB + DB concrete struct + closure pattern

本项目采用 **concrete struct + closure 模式**（见 ADR-010），通过 `MockDB` 和 `MockDBBuilder` 提供等效的测试隔离能力：

#### MockDB (预设全量结果)

```moonbit
let mock = MockDB::new(
  Ok(0L),                     // exec result
  Ok(0L),                     // execrows result
  Ok(RowIter::new(fn() { ... })), // query result
  Ok(Row::new(...)),          // query_row result
  Ok(0L), Ok(0L), Ok(0L),    // copyfrom/batch/execlastid
  None,                        // no transaction support
)
let db = mock.build()  // -> DB
```

#### MockDBBuilder (按 SQL 精确匹配)

```moonbit
let db = MockDBBuilder::new()
  .register_query(
    "SELECT id, name FROM users WHERE id = $1",
    Ok(single_row_iter)
  )
  .register_exec(
    "INSERT INTO users (name) VALUES ($1)",
    Ok(1L)
  )
  .build()  // -> DB
```

#### 生成的查询函数天然兼容

由于所有生成的查询函数接受 `DB` 参数（connection-oriented functional API），constraint 闭合模式与 MockDB 无缝协作：

```moonbit
// 生成代码
fn query_users_by_id(db: DB, id: Int64) -> Result[Option[User], DBError] {
  let sql = "SELECT id, name FROM users WHERE id = $1"
  db.query_row(sql, [Int64(id)])?; let row = ...; row.decode()
}

// 测试 — 直接传入 MockDB 构建的 DB
test "query users by id" {
  let db = MockDBBuilder::new()
    .register_query_row("SELECT id, name FROM users WHERE id = $1", Ok(mock_row))
    .build()
  let result = query_users_by_id(db, 42L)
  // assert result
}
```

### 与 Go emit_interface 方案的对比

| 特性 | sqlc-gen-go (interface) | 本项目 (MockDB) |
|------|------------------------|-----------------|
| Mock 注入方式 | 实现 `Querier` 接口 | 构建 `DB` struct |
| 运行时开销 | Go interface dispatch | closure 间接调用 |
| SQL 级精确匹配 | 外部 mock 库实现 | MockDBBuilder 内置 |
| 编译期类型检查 | interface contract | struct field 类型检查 |
| 学习曲线 | 需理解 Go interface | 直接使用 MockDB API |
| 灵活性 | 任意 mock 框架 | MockDB / MockDBBuilder |
| 事务模拟 | 需额外实现 tx 接口 | with_tx() 方法 |

## 未来恢复路径

当 MoonBit 工具链满足以下条件之一时，可重新评估 `emit_interface` 的可行性：

1. **MoonBit 支持 trait object** (`dyn Trait` 语法) — 生成 `trait Querier { ... }` 并让 `DB` 实现它，用户可以声明 `fn handle(db: Querier)`
2. **MoonBit 支持泛型 trait 方法** — 允许 trait 方法有独立泛型参数（例如 `fn query[T](self, sql: String) -> Result[T, DBError]`）
3. **MoonBit 支持 trait 继承/组合** — 允许组合多个 trait 形成一个更大的 trait

恢复后的实现路径：

```moonbit
// 未来 MoonBit 可能支持的生成代码 (示意)
pub trait Querier {
  query_users_by_id(Self, Int64) -> Result[Option[User], DBError]
  query_users_by_name(Self, String) -> Result[Array[User], DBError]
}

// DB 实现 Querier
impl Querier for DB { ... }

// 用户代码 — mock 可使用任何实现 Querier 的类型
fn process_users(db: Querier) -> Result[Array[User], DBError] {
  db.query_users_by_id(42L)
}
```

## 理由

- **不等待工具链**: MoonBit 0.1 的 trait 支持有限，等待工具链更新意味着测试基础设施长期缺失
- **MockDB 提供等效能力**: MockDB + MockDBBuilder 覆盖了 emit_interface 的核心用例 — 数据库依赖隔离和测试 mock
- **接口抽象非必须**: 生成的代码采用 connection-oriented functional API，不引入 Repository/DI 层，`DB` struct 本身已达到测试可替换性
- **低切换成本**: 生成代码接受 `DB` 参数（非 impl Trait），未来切换到 trait-based 抽象时，现有生成的函数签名无需修改

## 后果

### 正向

- MockDB + MockDBBuilder 提供了完整的测试 mock 基础设施，覆盖率与 Go 生态的 `emit_interface` 相当
- 用户无需理解 MoonBit trait 系统即可编写数据库测试
- 基于 closure 的 mock 在运行时与真实 DB 调用路径一致（无额外的 dispatch 层）
- 当 MoonBit 工具链成熟后，可以平滑过渡到 trait-based 方案（生成代码的 `DB` 参数签名兼容）

### 负向

- 用户无法像 Go 生态那样通过实现 `Querier` 接口来自定义 mock（但 MockDBBuilder 提供了替代的注册式 API）
- 未来引入 trait-based 接口时需要额外的 codegen 选项（`emit_interface`）和生成路径
- 与 sqlc-gen-go 的 `emit_interface` 配置模式不兼容 — 用户无法复用 sqlc-gen-go 的 `emit_interface` 配置
