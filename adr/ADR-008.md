# ADR-008: Native WASI I/O via Inline WAT FFI

- **状态**: 已接受
- **日期**: 2026-05-17
- **提出者**: 架构决策

## 背景

最初设计采用 WAT shim 后处理合并方案（ADR-007）：`moon build --target wasm` 产生 WASM 后，通过 `scripts/merge-shim.ps1` 将手写 `shim/wasi_shim.wat` 文本合并到 MoonBit 生成的 WAT 中，再由 `wat2wasm` 编译为最终 `plugin.wasm`。

该方案在实践中暴露了三个问题：
1. **PS 版本兼容**: merge-shim.ps1 使用 `Select-String -Raw`（PS 6+），PS 5.1 不支持
2. **合并编译失败**: linked.wat 中包含 MoonBit 已定义的 `$moonbit.bytes_make_raw`，stub 段重复定义导致冲突；MoonBit core 包的 global 定义在合并时未正确包含
3. **维护成本**: 后处理步骤增加构建流程复杂度，MoonBit 工具链更新后需同步调整 shim

## 决策

弃用 WAT shim 后处理方案，改为 **MoonBit 原生 I/O 方案**：

- 利用 MoonBit `--target wasm` 的 `= "module" "name"` 语法直接导入 WASI `fd_read`/`fd_write`
- 利用内联 WAT FFI `= "(func ...)"` 执行原始内存操作（iovec 构造、Bytes 数据指针计算）
- I/O 协议循环实现在纯 MoonBit 源码中（`plugin/wasi_io.mbt`），`main` 函数直接调用 `run_io_loop()`
- 构建流程简化为单一的 `moon build --target wasm`，无后处理步骤

## 理由

- **原生集成**: MoonBit 工具链直接支持 WASI FFI，无需外部脚本或 WAT 合并
- **可维护性**: I/O 逻辑在 MoonBit 源码中，可用 MoonBit 类型系统和测试框架覆盖
- **可测试性**: `process_message` 保持纯函数（195 测试覆盖），`run_io_loop` 仅在 WASM 入口生效
- **工具链兼容**: MoonBit 更新后，I/O 代码自然适配新版本编译器，无需迁移 shim
- **体积可控**: 170KB WASM 二进制（含全部代码生成逻辑 + I/O 协议层）

## 后果

- 正向：`moon build --target wasm` 一步完成，构建过程可复现
- 正向：iovec 结构体（12 字节）固定在 [1024,1035] 预留区，数据缓冲区由 GC `Bytes::new` 动态分配，无需手动内存管理
- 正向：WASM import 统一由 MoonBit 编译器管理（避免重复 import 冲突）
- 负向：WAT shim 参考设计归档到 `shim/archive/`，供未来 MoonBit 工具链更新后对比参考
