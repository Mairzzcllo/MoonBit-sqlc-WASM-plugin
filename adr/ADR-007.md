# ADR-007: WAT Shim ABI Bridge

- **状态**: 已接受
- **日期**: 2026-05-17
- **提出者**: 架构决策

## 背景

MoonBit sqlc WASM 插件需要实现 WASI `fd_read`/`fd_write` 调用以支持 sqlc (wasmtime v1.31.1) 的 stdin/stdout 帧协议。MoonBit 在 `--target wasm` 模式下：

- 不支持 wasm-gc（wasmtime v1.31.1 未实现 wasm-gc）
- 无法直接构造 WASI iovec 结构（8 字节：buf_ptr + buf_len）
- 无法获取 `Bytes` 对象的原始线性内存指针
- `#borrow`/`#owned` FFI 注解存在 parse error

## 可选方案

### A — 等待 MoonBit 更新 FFI

等待 MoonBit 上游修复 `#borrow`/`#owned` 解析错误或支持原始指针 API。

- 问题：无法控制上游时间线，MoonBit wasm backend 仍在快速变化中
- 风险：项目被 runtime blocker 卡死，不可控

### B — WAT shim ABI bridge ✅

在 MoonBit 编译产物之外，手写 WAT shim 层处理 iovec 构造和 raw memory 操作。

- 方案可行：MoonBit 已导出 `memory`，.data 初始段在 10000+，[1024, 1035] 区间空闲
- 侵入小：MoonBit 仅需修改 FFI 目标模块名和参数传递方式
- 风险低：不依赖上游生态，不修改 MoonBit runtime

### C — 升级 sqlc 支持 wasm-gc

推动 sqlc upstream 从 wasm32-wasi 迁移到 wasm-gc。

- 问题：sqlc 官方主支持 WASI，非 wasm-gc
- 风险：项目从"写 MoonBit sqlc plugin"变成"修 wasm-gc toolchain"，复杂度完全不同

## 决策

采用 **方案 B — WAT shim ABI bridge**。

## 架构

*初版设计：MoonBit 通过 `"shim"` FFI 调用 shim 层，shim 构造 iovec 后调用 WASI。*

**实际实现（P0-024 验证后修正）：MoonBit `--target wasm` FFI 不支持 reference types（Bytes/String），error 4042。改为反向方向：shim 包装入口点，通过直接 WAT 调用调用 MoonBit 内部函数。**

```
wasmtime(sqlc)              shim wrapper (post-merge)          MoonBit (pure)
      │                             │                              │
      │  stdin: framed msg ────────►│                              │
      │                             │  fd_read → scratch buffer    │
      │                             │  alloc Bytes via moonbit     │
      │                             │    .bytes_make_raw           │
      │                             │  copy scratch → Bytes data   │
      │                             ├─── call process_message ────►│
      │                             │                              │  decode → process
      │                             │◄── return response Bytes ────│  → encode
      │                             │  read Bytes data + length    │
      │  stdout: framed resp ◄──────│  fd_write                    │
```

- MoonBit 侧：零 FFI 声明。暴露 `pub fn process_message(data: Bytes) -> Bytes`
- Shim 侧：post-merge 注入 MoonBit 模块，通过直接 WAT 函数调用调用 `moonbit.bytes_make_raw`、`moonbit.array_length` 和 MoonBit 的 `process_message`
- 保留内存：`[1024, 1035]` — iovec + rof_len；`[1036, 65535]` — scratch buffer

## 理由

- **风险最低**：只解决 ABI 兼容问题，不引入复杂工具链
- **侵入最小**：不修改 MoonBit runtime 或 allocator
- **边界清晰**：MoonBit 只负责 raw bytes (ptr, len)，WAT 层只负责 iovec
- **不依赖上游**：无第三方时间线风险
- **标准做法**：Zig、TinyGo、AssemblyScript 等 WASM 工具链在面对 runtime ABI 不稳定时都采用类似模式

## 后果

- 正向：WASM I/O 层不再依赖 MoonBit FFI 更新进度
- 正向：保留内存区间固定，便于调试和验证
- 正向：项目核心技术风险（codegen pipeline）已解除，只剩工程适配
- 负向：需要构建后处理步骤（shim 合并）
- 负向：手写 WAT 增加了少量的维护成本
- 注意：保留内存区间 [1024, 1035] 不可与 MoonBit 未来版本的数据段冲突，升级 MoonBit 时需验证
